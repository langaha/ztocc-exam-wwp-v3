# 《需求理解与假设说明》（工程落地版）

本文档把题目中“故意留白”的规则补全为一组可配置、可实现、可解释的默认值，并说明依据。后续实现需要把这些规则固化为：数据库配置项 + 后台可修改能力，而不是写死在 if/else。

## 0. 全局假设

- 单租户：V2 数据未体现租户字段，本实现假设单租户；若要多租户，需要在 V2 运单表与 V3 工单/扫描/库存表增加 `tenant_id`，并在鉴权后把 `tenant_id` 注入所有查询条件。
- 金额口径：题目里的“金额”用于审批分级与赔付金额，不强绑定“运单原始金额”。本实现中：
  - 运单快照可带 `estimated_amount`（若 V2 未提供则为空）
  - 工单上报/执行需要填写 `claim_amount`（理赔/追偿金额），以该金额驱动审批分级
- 与题目原文差异：数据库同实例同连接字符串，但 V3 仅使用 `v3_` 前缀表；V3 仍通过 HTTP 调 V2 获取运单数据，禁止 V3 直查 V2 业务表。

## 1) 分级审批金额阈值（可配置）

默认配置（单位：元），存入 V3 的 `v3_approval_rules` 表：

- 物流异常：
  - `claim_amount <= 500`：一级审批即可
  - `500 < claim_amount <= 2000`：二级审批
  - `claim_amount > 2000`：二级审批 + 强制备注“高额理赔”
- 品控异常（追偿方向）：
  - `claim_amount <= 1000`：一级审批即可
  - `claim_amount > 1000`：二级审批

依据：

- 物流理赔对客户体验更敏感，低额快速放行；高额需要二级把关。
- 品控追偿涉及供应商对账与证据链，整体阈值略高但仍需要分级控制风险。

## 2) 审批超时时长（可配置）

默认配置：

- 待审批（P1）：2 小时未处理 → 自动升级到二级审批中（P3）
- 一级审批中（P2）：4 小时未处理 → 自动升级到二级审批中（P3）
- 二级审批中（P3）：8 小时未处理 → 自动驳回（DR）并关闭工单（记录“超时驳回”审批记录）

依据：

- 运营侧需要 SLA，避免长期挂起；二级审批默认给更长处理窗口。
- 二级长期不处理通常意味着决策不可达，选择自动驳回以避免资金/库存长时间冻结。

## 3) 拒绝后的重新提交次数上限（可配置）

默认配置：

- 允许重新提交 2 次（即最多 3 轮：首次 + 2 次重提）
- 超过次数上限：自动升级到二级审批中（P3），并在工单审计里记录“多次重提升级”

依据：

- 允许操作人补充证据/修正填写错误；但避免无限循环占用审批资源。

## 4) 物流异常类型映射（下游动作）

默认映射（可在 V3 以配置表方式维护，初始内置如下）：

- 丢件：理赔 + 重新发货（扣减库存）+ 关闭原批次锁（如存在）
- 破损：理赔（可选退货入库，按实际是否回收）+ 重新发货
- 客户拒收：退货入库（加库存）+（可选）不理赔
- 超时未签收：先进入“执行中-调查”子状态（仍属于执行中）；若确认丢件则按丢件处理
- 收货地址错误：重新发货（扣减库存）；默认不理赔

依据：

- 行业常识：地址错误/拒收通常更偏履约调整而非赔付；丢件/破损更偏资金赔付与补发。

## 5) 角色权限划分

默认角色：

- `reporter`：上报异常/提交重提材料
- `approver_l1`：一级审批
- `approver_l2`：二级审批
- `qc_supervisor`：品控主管（允许误判快速放行）
- `admin`：后台配置（规则/用户/种子数据/监控）

强制规则：

- 上报人不能审批自己提交的工单（一级/二级都不允许）
- 误判快速放行仅 `qc_supervisor` 可操作（且必须填写原因，留痕）
- 禁用账号不能执行任何审批/放行动作；其待办需由后台任务转交/升级（见状态机文档）

## 6) V2 数据同步频率与一致性策略

默认策略：

- 关键动作实时校验：创建工单/扫描录入时，必须实时调用 V2 校验运单存在与 SKU 归属（若 V2 不可用则禁止创建/扫描，给出明确错误提示）
- 展示可降级：工单详情页展示运单信息时，优先实时拉取；若 V2 超时/不可用则降级为本地快照，并明确标注“数据同步于 XX 时间”
- 增量同步（可选但建议做）：定时任务每 10 分钟同步最近 24 小时出现过的运单（以减少详情页频繁实时拉取压力）

## 7) 品控暂扣超时时长（独立于审批超时）

默认配置：

- 品控暂扣（批次锁定）30 分钟未关闭 → 强制升级至二级审批中（P3），并在工单审计里记录“压仓超时升级”

依据：

- 压仓直接产生仓内运营成本，应该显著短于审批超时；30 分钟通常足够现场复核。

## 8) 品控规则触发阈值（可配置）

默认内置规则样例（所有阈值均来自 `v3_qc_rules` 表，可后台调整）：

- 数量不符：
  - `abs(scanned_qty - expected_qty) / expected_qty >= 0.02` → 异常（严重度=低，默认进入一级审批）
  - `>= 0.10` → 异常（严重度=高，默认直接进入二级审批）
- 外观破损：
  - `damage_level >= 2` → 异常（一级审批）
  - `damage_level >= 4` → 异常（二级审批）
- 规格不符/标签错误/批次异常：默认直接异常（一级审批），由品控主管可在后台把某些子类提升为二级

依据：

- 小比例差异可能是扫描误差或装箱波动；高比例差异风险更大，应提高审批等级。

## 9) 品控主管角色权限边界

默认边界：

- `qc_supervisor` 可以：
  - 对品控工单执行“误判快速放行”（关闭工单 + 解锁批次），必须填写原因并写入审批/审计记录
  - 查看全部扫描记录、命中规则、暂扣原因
- `qc_supervisor` 不等于审批人：
  - 默认不授予 `approver_l1/approver_l2`，除非管理员显式给同一账号配置多角色
  - 即使同一账号拥有审批角色，也仍然受“不能审批自己上报的工单”的约束

## 10) 如果能问产品经理，会问的问题清单（加分项）

- 运单金额口径：审批阈值基于“理赔金额”还是“运单货值/订单金额”？是否需要从 ERP/WMS 获取？
- 品控暂扣超时后的策略：一定要强制二级审批吗？是否允许“自动放行但标记追踪”？
- 物流异常的“超时未签收”调查期：需要几天？期间是否冻结库存/赔付预算？
- 快速放行的合规要求：是否需要二次复核/主管签字？是否需要上传图片/视频证据？
- 赔付与对账：赔付记录是否需要对接财务系统？对账周期与状态有哪些？
